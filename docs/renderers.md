# Renderers

A renderer takes an `agent-policy.yaml` and produces one instruction file. The built-in renderers are:

| Renderer id | Default path | Style |
|---|---|---|
| `agents_md` | `AGENTS.md` | Full canonical document. |
| `claude_md` | `CLAUDE.md` | Concise mirror with a preamble pointing to `AGENTS.md`. |
| `copilot_instructions` | `.github/copilot-instructions.md` | Concise, plain-text style, no preamble. |

## Shared output structure

Every renderer emits:

```
<banner>
# <project.name>
<project.description, if set>

## <section 1 title>
<section 1 body>

## <section 2 title>
<section 2 body>

...

## Attribution
<attribution policy summary>
```

The banner is two HTML comment lines:

```
<!-- Generated by agent-policy. Do not edit directly. Edit agent-policy.yaml instead. -->
<!-- agent-policy:hash=<16-char sha256 prefix> -->
```

The hash is a SHA-256 of the rendered body with the banner stripped, so it is reproducible across machines and commits.

## `concise` filtering

When a renderer has `concise: true`, it includes a section only if:

1. The section has `pinned: true`, **or**
2. The section's `renderers` array explicitly lists this renderer's id.

If `concise: false` (default), every section is included regardless of its `renderers` field.

This lets one `agent-policy.yaml` drive a long `AGENTS.md` and short `CLAUDE.md` / Copilot files from the same source.

## Determinism contract

- Section order matches declaration order in YAML. No implicit sorting.
- Line endings are `\n`. A trailing newline is always present.
- No timestamps, host info, or user info in output. Same input → same bytes.
- Verified by a CI test that renders the full example twice and asserts byte-equality.

## Generated banner guard

`sync` refuses to overwrite a file that doesn't start with the banner line, unless `--force` is passed. This protects a maintainer who hand-wrote an `AGENTS.md` before adopting the tool: they get a clear error instead of silent clobbering.

## Adding a new renderer

Renderers implement `Renderer`:

```ts
interface Renderer {
  id: RendererId;
  defaultPath: string;
  render(config: AgentPolicyConfig): string;
}
```

See [`src/renderers/agents-md.ts`](../src/renderers/agents-md.ts) for the simplest implementation and [`src/renderers/claude-md.ts`](../src/renderers/claude-md.ts) for one that uses `concise` filtering.

To wire a new renderer:

1. Add its id to the `RendererId` union in [`src/config/types.ts`](../src/config/types.ts).
2. Add a `$defs/renderer` property for it in [`schemas/agent-policy.schema.json`](../schemas/agent-policy.schema.json).
3. Add it to the `enum` list under `$defs/section/properties/renderers/items`.
4. Create the renderer file in `src/renderers/`.
5. Register it in [`src/renderers/index.ts`](../src/renderers/index.ts).
6. Add a snapshot test under `tests/`.

The existing registry, determinism test, and `sync`/`sync --check` behavior pick the new renderer up automatically.
